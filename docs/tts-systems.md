### MediaTek-Research/BreezyVoice

- **Repo:** https://github.com/mtkresearch/BreezyVoice
- **License:** Apache-2.0 — declared in both the HF model card (cardData license: apache-2.0) and the GitHub LICENSE file (Apache License 2.0). Code is partially derived from Alibaba CosyVoice (also Apache-2.0). Note: the optional ttsfrd resources (CosyVoice-ttsfrd/, festival dicts) shipped in the HF repo carry their own upstream licenses, but they are not required (WeTextProcessing fallback).
- **Runnable:** moderate (confidence: high)
- **GPU / disk / sample rate:** ~3-5 GB. The LLM is CosyVoice-300M scale; total fp32 weights ~2.6 GB load to GPU. Comfortably fits any T4/L4/A10 (16 GB). README explicitly states it also runs on CPU (swap onnxruntime-gpu -> onnxruntime). / ~3.5 GB of model weights: llm.pt 1.24 GB, flow.pt 0.42 GB, hift.pt 0.082 GB, speech_tokenizer_v1.onnx 0.52 GB, flow.decoder.estimator.fp32.onnx 0.33 GB, campplus.onnx 28 MB, spk2info.pt 1.3 MB. Plus ttsfrd festival resources in the HF repo (only if ttsfrd installed), the G2PW model (~a few hundred MB, downloaded by g2pw on first use), and whisper-base (~150 MB, only if you let it auto-transcribe the reference). Budget ~5 GB. / 22050 Hz / trust_remote_code=False

**What it is.** BreezyVoice is a zero-shot voice-cloning TTS for Taiwanese Mandarin, derived from Alibaba's CosyVoice-300M and fine-tuned by MediaTek Research. Pipeline: an S3 speech tokenizer (ONNX) extracts semantic speech tokens from a reference clip, a Transformer LLM (~300M, llm.pt) autoregressively predicts speech tokens from text + speaker prompt, an OT-CFM flow-matching model (flow.pt) converts tokens to mel, and a HiFi-GAN vocoder (hift.pt) produces 22050 Hz audio. Its distinctive feature is a G2PW grapheme-to-phoneme frontend that auto-annotates rare/polyphonic characters with 注音 (Bopomofo) and accepts inline manual Bopomofo corrections for exact pronunciation/tone control. Text (Chinese chars + Bopomofo annotations + English) is tokenized with the Whisper multilingual tokenizer and fed to the fine-tuned LLM.

**Phonetic control** (pinyin=❌ zhuyin=✅ phoneme/ipa=❌, confidence=high):

YES — per-character pronunciation/tone override via inline 注音 (Bopomofo/Zhuyin). Place [:<bopomofo><tone-digit>] IMMEDIATELY AFTER the character you want to force. Tone is an Arabic digit 1-5 (1=ˉ first, 2=ˊ, 3=ˇ, 4=ˋ, 5=˙ neutral), matching g2pw output. How it flows through the code: (1) text_normalize_new() in single_inference.py protects any [...] span from text normalization (split_by_brackets regex \[(.*?)\] keeps bracket content verbatim and re-joins it); (2) get_bopomofo_rare() runs G2PWConverter over the text and AUTO-inserts char+'[:'+bopomofo+']' for rare chars (corpus freq < 500) or polyphones whose predicted reading is non-default — and it SKIPS any char already followed by '[' (checks next_t_char != '['), so your MANUAL annotations win; (3) the resulting string (Han chars + Bopomofo glyphs + brackets + digits + Latin letters) is encoded by whisper.tokenizer.get_tokenizer(multilingual=True) with allowed_special='all' and passed to the fine-tuned LLM, which was trained to map these Bopomofo annotations to pronunciation. NOTE: control is Bopomofo ONLY — there is no Pinyin, IPA, SSML, or arbitrary-phoneme input path in BreezyVoice's pipeline (the Pinyin-capable ttsfrd frontend is an unused CosyVoice fallback). To force a proper noun you annotate each character with its Bopomofo+tone. There is also no word-level lexicon override; control is character-by-character inline.

- _Example:_ `Single override (from README): 今天天氣真好[:ㄏㄠ3]  |  Proper noun 台積電 (tái-jī-diàn), forcing every syllable: 台[:ㄊㄞ2]積[:ㄐㄧ1]電[:ㄉㄧㄢ4]  |  forcing just the polyphone 積: 台積[:ㄐㄧ1]電`

**zh-TW support.** Strong / core design goal. Paper title: 'Adapting TTS for Taiwanese Mandarin with Enhanced Polyphone Disambiguation'. The whole pipeline targets Taiwan readings: Traditional Chinese (OpenCC s2t), the G2PW Bopomofo G2P, and a per-character char2phn table of Taiwan-standard Bopomofo readings (utils/word_utils.py). Sample data uses Taiwan content/toponyms (信義快速道路, 臺北, 聯發創新基地) and example transcripts are Traditional Chinese. README claims it 'outperforms competing commercial services in terms of naturalness'. So it is purpose-built for the Taiwan accent rather than mainland; mainland-accent drift is unlikely given the zh-TW fine-tune and Bopomofo frontend. Caveat: in their own code-switching table the 'Toponyms' category scored low (3), and accent fidelity also depends on the reference speaker clip you provide. Evidence: README tables + arXiv:2501.17790 abstract ('polyphone disambiguation' for Taiwanese Mandarin) + code (opencc s2t, g2pw, char2phn). Confidence high that target accent is zh-TW.

**Code-switch.** Supported within a single utterance. README has a dedicated Chinese-English code-switching benchmark where BreezyVoice tops most categories (General Words 8, Entities 9, Abbreviations 9; weaker on Toponyms 3, Full Sentences 7) vs four competitors, and the paper states 'superior performance in both general and code-switching contexts'. The shipped example run_single_inference.sh synthesizes a mixed sentence: '歡迎使用聯發創新基地 BreezyVoice 模型。' (English token embedded in Chinese). Mechanism: text is tokenized by the Whisper multilingual tokenizer (allowed_special='all'), so Latin/English subwords are valid tokens passed straight to the LLM; text_normalize_new handles Chinese vs English segments. Evidence: README code-switching table + run_single_inference.sh + arXiv:2501.17790. Confidence high.

**Voice mode.** type=zero-shot-clone, needs_reference_audio=True, needs_reference_text=True.

Primary mode is zero-shot voice cloning: you MUST supply a reference speaker wav (speaker_prompt_audio_path, loaded at 16 kHz). A reference transcript (speaker_prompt_text_transcription) is 'highly recommended' and used as the LLM prompt_text; it is technically OPTIONAL — if omitted, single_inference.transcribe_audio() auto-transcribes with openai/whisper-base + OpenCC s2t (Traditional). The reference text is itself run through get_bopomofo_rare so its rare chars get Bopomofo too. There is NO voice-design-by-text. A CosyVoice preset-speaker path exists in code (frontend_sft + spk2info.pt + list_avaliable_spks / inference_sft) but the BreezyVoice scripts (single_inference.py, batch_inference.py, api.py) only use the zero-shot clone path — treat preset speakers as effectively unused.

**Minimal inference.**
```python
import os; os.environ['PYTHONUTF8']='1'
# run from inside a clone of github.com/mtkresearch/BreezyVoice
import torchaudio
from g2pw import G2PWConverter
from cosyvoice.utils.file_utils import load_wav
from single_inference import CustomCosyVoice, get_bopomofo_rare

# 1) load model (weights auto-download from HF on first run; ~2.6GB .pt + ~0.9GB onnx)
cosyvoice = CustomCosyVoice('MediaTek-Research/BreezyVoice')
bopomofo = G2PWConverter()

# 2) zero-shot speaker = reference wav + its transcript (transcript optional but recommended)
prompt_wav = load_wav('./data/example.wav', 16000)
prompt_text = '在密碼學中，加密是將明文資訊改變為難以讀取的密文內容。'

# 3) target: Chinese + an English word (code-switch) + manual Bopomofo override on a proper noun
text = '歡迎使用 BreezyVoice，台積[:ㄐㄧ1]電是台灣的公司。'

# 4) normalize -> auto-annotate rare/polyphone chars with 注音 (keeps your manual [:...]) -> synth
prompt_text = cosyvoice.frontend.text_normalize_new(prompt_text, split=False)
text = cosyvoice.frontend.text_normalize_new(text, split=False)
prompt_text = get_bopomofo_rare(prompt_text, bopomofo)
text = get_bopomofo_rare(text, bopomofo)
out = cosyvoice.inference_zero_shot_no_normalize(text, prompt_text, prompt_wav)

torchaudio.save('out.wav', out['tts_speech'], 22050)  # 22050 Hz
```

### ResembleAI/chatterbox (Chatterbox Multilingual V3, T3 checkpoint t3_mtl23ls_v3.safetensors; language_id="zh"). Dedicated Chinese finetune: ResembleAI/Chatterbox-Multilingual-zh-cmn.

- **Repo:** https://github.com/resemble-ai/chatterbox
- **License:** MIT (model card front-matter 'license: mit' on ResembleAI/chatterbox and the zh-cmn finetune; repo pyproject license={file='LICENSE'}; code acknowledges Llama 3 / CosyVoice / HiFTNet). Note: all generated audio is embedded with a Perth neural watermark by default.
- **Runnable:** easy (confidence: high)
- **GPU / disk / sample rate:** ~6-8 GB recommended in fp32 (weights ~3 GB: 2.1 GB T3 + ~1 GB S3Gen + 6 MB VE, plus activations). Runs comfortably on a 16 GB T4/L4 or 24 GB A10G/L4; CPU/MPS paths exist but are slow. (VRAM figure estimated from weight sizes + model size, not benchmarked.) / ~3.2 GB download for V3-zh: t3_mtl23ls_v3.safetensors 2.14 GB + s3gen.pt 1.06 GB + ve.pt 5.7 MB + conds.pt 0.1 MB + grapheme tokenizer 0.07 MB + Cangjie5_TC.json 1.9 MB. (Note: from_pretrained for V3 still downloads the V2 s3gen.pt, not s3gen_v3.) / 24000 (S3GEN_SR; reference audio is internally resampled to 16000 for the speaker encoder) Hz / trust_remote_code=False

**What it is.** Chatterbox Multilingual V3 is Resemble AI's open-source (MIT) 0.5B zero-shot voice-cloning TTS covering 23 languages including Chinese (language_id="zh"). Architecture: a Llama-520M "T3" text-to-speech-token model + a CosyVoice-style "S3Gen" flow-matching/HiFiGAN decoder (24 kHz output) + a voice-encoder for speaker embedding; all outputs carry a Perth neural watermark. Text is fed as graphemes via a BPE tokenizer with a per-language frontend; for Chinese the frontend (a) word-segments with pkuseg and (b) replaces each Han character with its Cangjie5 glyph-decomposition codes ([cj_*] tokens) — i.e. shape-based, NOT phonetic, so there is no built-in tone/pronunciation control. You drive it with model.generate(text, language_id, audio_prompt_path=...).

**Phonetic control** (pinyin=❌ zhuyin=❌ phoneme/ipa=❌, confidence=medium):

NO documented or wired phonetic-control API exists. The only inference entry point is ChatterboxMultilingualTTS.generate(text, language_id, ...); there is no pronunciation/SSML/phoneme argument. For language_id='zh' the text frontend (src/chatterbox/models/tokenizers/tokenizer.py -> MTLTokenizer.encode -> ChineseCangjieConverter) word-segments with pkuseg and then replaces every Han character (Unicode category 'Lo') with its CANGJIE5 GLYPH-decomposition codes (e.g. characters become sequences of [cj_a]..[cj_z]/[cj_0]..[cj_9] + [cj_.]). Cangjie is purely shape/orthography-based, so homographs with different readings map to the SAME tokens and there is NO way for the caller to force a tone/reading. Pronunciation is learned implicitly from glyph+context; an open HF discussion (#24 'IPA?') asking for direct phonetic input is unanswered by maintainers. EXPERIMENTAL (undocumented, UNVERIFIED by us) levers found only by reading the tokenizer JSON: the shared vocab (grapheme_mtl_merged_expanded_v1.json, 2454 tokens, BPE) DOES contain (1) the full Bopomofo/Zhuyin block U+3105-U+312F plus tone marks ˊ(U+02CA) ˇ(U+02C7) ˋ(U+02CB), and (2) a dedicated [ipa] and [end_of_label] control-token pair plus a coherent 56-symbol IPA inventory (incl. Mandarin retroflex/alveolo-palatal ɕ ʂ ʐ ʈ ɤ ɥ and English stress-marked vowels ˈiː ˌeɪ dʒ tʃ). Because the Cangjie converter passes through any non-Han character unchanged, manually inserting Zhuyin (with tone marks) or an [ipa]...[end_of_label] span into the zh input string survives lower()+NFKD normalization and tokenizes to those exact vocab tokens, physically reaching the model. WHETHER THE MODEL ACTUALLY HONORS THEM IS UNVERIFIED (would require running the model, out of scope); it may have been a training scheme or just a defensively-expanded vocab. Note the neutral-tone mark ˙(U+02D9) is destroyed by NFKD, and the IPA inventory lacks Mandarin lexical-tone diacritics, so Zhuyin+tone marks is the more tone-complete candidate. RECOMMENDATION: empirically A/B test Zhuyin-injection and the [ipa] span on a few proper nouns before relying on either.

- _Example:_ `Default (NO control): model.generate('台積電的營收創新高', language_id='zh')  ||  EXPERIMENTAL Zhuyin injection (tone marks survive, in-vocab; UNVERIFIED): model.generate('ㄊㄞˊ ㄐㄧ ㄉㄧㄢˋ 的營收創新高', language_id='zh')  ||  EXPERIMENTAL IPA span (tokens exist as [ipa]/[end_of_label]; UNVERIFIED): model.generate('[ipa]tɕí[end_of_label]積電', language_id='zh')`

**zh-TW support.** No Taiwan-specific (zh-TW) model or accent option. Only a generic language_id='zh' in the multilingual model and a dedicated Mandarin finetune 'ResembleAI/Chatterbox-Multilingual-zh-cmn' (locale zh-CMN, Chatterbox lang id 'zh') — 'cmn' = standard Mandarin, no Taiwan variant. ORTHOGRAPHY: Traditional characters (used in Taiwan) ARE covered because the Chinese frontend uses Cangjie5_TC.json, a Traditional-Chinese Cangjie glyph table (unmapped chars fall through unchanged), so Traditional input is accepted. ACCENT: comes from training data labelled zh/cmn, i.e. standard/likely-mainland Mandarin; there is NO Taiwan-accent control and the Cangjie glyph frontend carries no regional pronunciation info. The README notes accent can be inherited from the reference clip ('set cfg_weight to 0' to mitigate), so a Taiwan-accented reference clip MAY transfer some Taiwan timbre/accent, but underlying phoneme realization stays from the base model (unverified). Community feedback reports mispronunciation of less-common Chinese vocabulary and CJK speaking too fast at default settings. Net: expect generic/mainland-leaning Mandarin, partial accent transfer via reference only. Confidence low-medium (no zh-TW evidence in primary sources).

**Code-switch.** Supported and officially demonstrated. The repo's own multilingual demo (multilingual_app.py LANGUAGE_CONFIG['zh']) uses a Chinese sentence containing the English word 'YouTube' within one utterance. Mechanism: in the zh frontend only Han characters (category 'Lo') are converted to Cangjie [cj_*] tokens; Latin letters/digits pass through untouched and are tokenized as ordinary BPE Latin tokens, so a single 'zh' utterance can mix Han + English words and brand names. Quality of the code-switched English (and its pronunciation) was not verified by us. Confidence medium.

**Voice mode.** type=zero-shot-clone, needs_reference_audio=False, needs_reference_text=False.

Zero-shot voice cloning from a single reference WAV passed as audio_prompt_path (prepare_conditionals derives speaker embedding + prompt speech tokens; NO reference transcript is needed). A reference is OPTIONAL: a built-in default voice ships as conds.pt, so generate() works with no reference at all. There is no preset speaker library beyond that one built-in voice and no text-based voice design. Expressive controls: exaggeration (emotion intensity, default 0.5) and cfg_weight (default 0.5; set to 0.0 to stop the reference clip's language accent leaking into other-language output). A separate voice-conversion path exists (vc.py / example_vc.py).

**What it is.** Fun-CosyVoice 3.0 (0.5B) is Alibaba/FunAudioLLM's LLM-based zero-shot multilingual TTS. A Qwen2-0.5B backbone (shipped as CosyVoice-BlankEN) autoregressively predicts semantic speech tokens from text, a causal masked DiT flow-matching decoder turns tokens into mel, and a causal HiFi-GAN (hift) vocodes to 24 kHz audio; a CAM++ ONNX model extracts the speaker embedding and an ONNX speech tokenizer encodes the reference audio. It is end-to-end with no traditional g2p frontend (numbers/symbols normalized by wetext or optional ttsfrd). Crucially it adds "pronunciation inpainting": the caller can force pronunciation of any word by inserting bracketed Pinyin (tone as a Unicode diacritic) or English CMU/ARPAbet phoneme tokens directly in the text.

**Phonetic control** (pinyin=✅ zhuyin=❌ phoneme/ipa=✅, confidence=high):

YES — explicit, trained 'Pronunciation Inpainting'. You override pronunciation by REPLACING the problematic character(s) in the input text with bracketed phoneme tokens. Chinese: write each syllable as [initial][final], where the final token carries the TONE as a Unicode diacritic (tone1 macron ā, tone2 acute á, tone3 caron ǎ, tone4 grave à, neutral = bare vowel). Multi-letter initials/finals are single tokens (e.g. [zh],[ch],[sh],[iàn],[uáng]); ü is written 'v' (e.g. [vè],[ǘ]). English: a sequence of bracketed CMU/ARPAbet phones with stress digits 0/1/2 (e.g. [IH1][N][V][AH0][L][IH0][D]). These exact bracket strings are registered in cosyvoice/tokenizer/tokenizer.py (CosyVoice3Tokenizer.special_tokens, added via tokenizer.add_special_tokens to the Qwen2 tokenizer), so each maps to a dedicated trained token id. They pass through text normalization untouched because frontend_utils.remove_bracket strips only full-width （）【】 and backticks, NOT ASCII []. NOT supported: numeric pinyin like 'tai2'/'ji1' (tone must be the diacritic form), Zhuyin/Bopomofo (no ㄅㄆㄇ tokens exist), and IPA/SSML phoneme tags. So your 'tái jī diàn' must be converted to diacritic pinyin and split into [initial][final] tokens.

- _Example:_ `我在[t][ái][j][ī][d][iàn]工作，它是 the leading foundry.  (forces 台积电 = tái jī diàn; English word read natively. Polyphone example from docs: 报道[j][ǐ]予好评 forces 给 -> jǐ; 美[d][ì]空调 forces 的 -> dì. English example: I was treated as an [IH1][N][V][AH0][L][IH0][D].)`

**zh-TW support.** Not explicitly supported; defaults to mainland Putonghua. Evidence: cosyvoice/utils/common.py instruct_list enumerates only mainland accents (Cantonese/Guangdong, Dongbei, Gansu, Guizhou, Henan, Hubei, Hunan, Jiangxi, Minnan, Ningxia, Shanxi, Shaanxi, Shandong, Shanghai, Sichuan, Tianjin, Yunnan) — there is NO Taiwan-Mandarin / 台湾国语 instruction. README, the CV3 paper (arXiv:2505.17589), and the demo page never mention Taiwan/zh-TW; the '18 Chinese dialects' are all PRC. Pinyin inpainting uses the PRC pinyin standard (diacritic finals), so default polyphone readings follow mainland norms (e.g. 和 = hé, not Taiwan hàn). Taiwan accent is only achievable indirectly by zero-shot cloning from a Taiwan-accented reference wav (transfers timbre+accent), and even then Taiwan-specific lexical tone readings would need per-word pinyin inpainting. Confidence: medium (negative evidence from primary sources, no positive zh-TW claim anywhere). Minnan (闽南话) support is Hokkien dialect, not Taiwan-accented Mandarin.

**Code-switch.** Supported within a single utterance. Evidence: official demo page (funaudiollm.github.io/cosyvoice3) shows the mixed sentence '我买了一台美的空调and wind the clock.' (Chinese+English in one sentence) plus zh+Japanese and en+Korean mixes; README/paper stress multilingual + cross-lingual zero-shot across 9 languages. Mechanism: a single multilingual Qwen2 tokenizer (CosyVoice-BlankEN vocab) handles mixed scripts directly; text_normalize() only routes whole-segment TN by contains_chinese but does not block embedded English. For reliable code-switch the docs favor inference_cross_lingual (no reference transcript) or inference_zero_shot; embedded English words/proper nouns can additionally be pinned with CMU phoneme inpainting (e.g. [IH1][N][V][AH0][L][IH0][D]). Confidence: high.

**Voice mode.** type=zero-shot-clone, needs_reference_audio=True, needs_reference_text=True.

Reference audio is mandatory to produce any output — no built-in preset speakers ship (the HF repo has NO spk2info.pt, so list_available_spks() is empty and inference_sft is unusable). Three modes, all clone from a prompt wav (<=30s, internally resampled to 16k/24k): (1) inference_zero_shot(tts_text, prompt_text, prompt_wav) — best quality, ALSO requires prompt_text = '<system prompt><|endofprompt|>' + the exact transcript of the reference wav. (2) inference_cross_lingual(tts_text, prompt_wav) — clones WITHOUT any transcript (recommended for cross-lingual / code-switch). (3) inference_instruct2(tts_text, instruct_text, prompt_wav) — natural-language style/dialect/emotion/speed control (e.g. '...请用广东话表达。<|endofprompt|>') but still needs a reference wav. No pure text-to-voice-design (timbre always comes from audio). You can cache a cloned speaker via add_zero_shot_spk()+save_spkinfo() and then reuse it by zero_shot_spk_id.

### openbmb/VoxCPM2

- **Repo:** https://github.com/OpenBMB/VoxCPM
- **License:** Apache-2.0 (free for commercial use). Confirmed on HF model card, GitHub LICENSE, and pyproject.toml.
- **Runnable:** easy (confidence: high)
- **GPU / disk / sample rate:** ~8 GB (bfloat16, per model card). RTF ~0.30 on RTX 4090, ~0.13 with Nano-vLLM. / ~5 GB for VoxCPM2 weights: model.safetensors 4.58 GB + audiovae.pth 0.38 GB + tokenizer.json 3.7 MB. Add ~100s of MB more if load_denoiser=True (ModelScope iic/speech_zipenhancer_ans_multiloss_16k_base). / 48000 (output; accepts 16 kHz reference audio, AudioVAE V2 super-resolves to 48 kHz) Hz / trust_remote_code=False

**What it is.** VoxCPM2 is OpenBMB's tokenizer-free, diffusion-autoregressive multilingual TTS model (2B params, MiniCPM-4 backbone + AudioVAE V2), supporting 30 languages and 9 Chinese dialects with 48 kHz output. It is "tokenizer-free" in the sense that it ingests raw text at the character level (a LlamaTokenizerFast that splits multi-character Chinese tokens into single chars) and maps text directly to continuous speech latents via LocEnc -> TSLM -> RALM -> LocDiT, with no external phoneme/discrete-token frontend. It generates speech via three modes: voice design from a text description, zero-shot voice cloning from a reference clip, and "Hi-Fi/Ultimate" cloning using reference audio plus its transcript. Distributed as the pip package `voxcpm`; one-line load via `VoxCPM.from_pretrained("openbmb/VoxCPM2")`.

**Phonetic control** (pinyin=✅ zhuyin=❌ phoneme/ipa=✅, confidence=medium):

Curly-brace phoneme injection, documented in the Usage Guide. To force pronunciation you (1) set normalize=False and (2) wrap each syllable in braces. Chinese: per-syllable Pinyin WITH a tone number, e.g. {ni3}{hao3}{shi4}{jie4} (tones 1-4, 5/0 = neutral). English: CMUDict/ARPAbet phonemes, e.g. {HH AH0 L OW1}. NO Zhuyin/Bopomofo and NO IPA support (issue #245 only asks about IPA fine-tuning; English uses ARPAbet, not IPA). Braces can be interleaved with normal Han characters, so you can override just one proper noun in a sentence. CRITICAL CAVEAT: this is NOT a deterministic g2p frontend - the repo's text frontend (src/voxcpm/utils/text_normalize.py, core.py, voxcpm2.py) contains ZERO curly-brace parsing; the braces are passed verbatim to the model, which only LEARNED to interpret Pinyin/ARPAbet annotations from training data. It is therefore a soft, probabilistic control. If text normalization is left on, the tone DIGIT is spoken literally (issue #172). And even with normalize=False, maintainers acknowledge polyphone/tone control 'is not very stable' on some voices and can yield the wrong reading (issue #172, reply by liuxin99). For a dataset of thousands of proper nouns needing exact tones, expect to use this syntax but verify outputs / regenerate, not 100% guaranteed.

- _Example:_ `model.generate(text='我在 {tai2}{ji1}{dian4} 上班。', normalize=False)  # forces 台積電 = tai2 ji1 dian4`

**zh-TW support.** Weak / not native. Default standard-Chinese text is rendered in MAINLAND Putonghua accent. The supported 'Chinese dialects' are 四川话/粤语/吴语/东北话/河南话/陕西话/山东话/天津话/闽南话 (Sichuanese, Cantonese, Wu, Northeastern, Henan, Shaanxi, Shandong, Tianjin, Min-Nan) - note Min-Nan/Hokkien is spoken in Taiwan but is a separate topolect, NOT Taiwan-accent Mandarin (台灣國語). No Taiwan-Mandarin examples appear in the model card, demo page, or docs. Best path to a Taiwan flavor is voice cloning from a Taiwan-accent reference clip: GitHub issue #321 confirms 'Ultimate/Hi-Fi cloning' carries the reference's accent (and a maintainer says regular cloning leans back toward the target-text/standard accent). Evidence: HF model card dialect list; issue #321; absence of zh-TW samples on openbmb.github.io/voxcpm2-demopage. Confidence medium.

**Code-switch.** Supported (multilingual model, no language tag needed - 'input text in any supported language directly'). The char-level tokenizer encodes both Han and Latin scripts in one stream, and the arXiv report markets smooth prosody 'on long-form content and code-mixed text'. Real-world zh-en mixing works (model card's own English examples embed the coined word 'VoxCPM2'; community spaces like indicvox-hindi-tamil-codeswitching-tts are built on it). CAVEAT: the demo page (openbmb.github.io/voxcpm2-demopage) shows cross-lingual voice transfer but NO explicit single-sentence zh-en code-switch sample, and the text normalizer (text_normalize.py) picks ONE language (zh if any Han char present, else en) for number/text normalization - the mixing happens at the model level, not the frontend. So zh-en code-switch within one utterance is supported but not separately benchmarked; quality should be spot-checked. Confidence medium.

**Voice mode.** type=mixed, needs_reference_audio=False, needs_reference_text=False.

Three modes, none strictly required for basic synthesis: (1) Voice Design (NO reference) - put a natural-language voice description in parentheses at the start of text, e.g. '(A young woman, gentle and sweet voice)Hello'. With no reference at all the model still synthesizes but picks a RANDOM timbre each call. (2) Controllable Voice Cloning - pass reference_wav_path only (5-30s clip); NO transcript needed; optional '(style)' instruction prefix to steer pace/emotion. (3) Hi-Fi / Ultimate Cloning - pass prompt_wav_path + prompt_text (the EXACT transcript of the prompt audio, ASR-recommended); for max similarity pass the same clip to both prompt_wav_path and reference_wav_path. So: reference audio is OPTIONAL; reference TRANSCRIPT is only needed for the highest-fidelity Hi-Fi mode (and disables the style instruction). reference_wav_path is VoxCPM2-only.

### OpenMOSS-Team/MOSS-TTS-Local-Transformer-v1.5

- **Repo:** https://github.com/OpenMOSS/MOSS-TTS
- **License:** Apache-2.0 (declared in the HF model card YAML front matter for both v1.5 and v1.0; the GitHub repo and MOSS-Audio-Tokenizer-v2 are also Apache-2.0).
- **Runnable:** moderate (confidence: high)
- **GPU / disk / sample rate:** Backbone safetensors ~9.1 GB (bf16) + MOSS-Audio-Tokenizer-v2 codec ~4.3 GB (bf16 compute; stored fp32 ~8.5 GB on disk) => ~13-14 GB weights, ~16-20 GB peak with KV cache for 4096 tokens. Both repos download on first run. / 48000 (native 48 kHz STEREO; processor.decode returns [channels, samples], save directly). NOTE v1.0 was 24 kHz mono. Hz / trust_remote_code=True

**What it is.** MOSS-TTS-Local-Transformer-v1.5 is the OpenMOSS team's 4B "MossTTSLocal" TTS model: a Qwen3-4B Transformer backbone (hidden 2560, 36 layers) that emits a per-frame block of 12 RVQ audio codes via a lightweight local/depth transformer, decoded by the separate MOSS-Audio-Tokenizer-v2 codec to native 48 kHz STEREO audio at a 12.5 Hz frame rate. It is loaded through standard HF AutoProcessor/AutoModel with trust_remote_code=True (arch moss_tts_local). It supports reference-less synthesis, zero-shot voice cloning, 31 languages with optional language tags, token-level duration control, explicit [pause X.Ys] markers, Chinese-English code-switching, and Pinyin/IPA pronunciation control. Crucially, the processor does NO g2p/text-normalization: the input text (including Pinyin and IPA) is passed unchanged through the Qwen3 BPE tokenizer, and the model was trained to render numbered-tone Pinyin and slash-wrapped IPA.

**Phonetic control** (pinyin=✅ zhuyin=❌ phoneme/ipa=✅, confidence=high):

Pronunciation is forced by writing the phonetic spelling DIRECTLY inside the `text` field of processor.build_user_message(text=...). There is NO g2p/frontend in the processor (confirmed by grep: processing_moss_tts.py has zero pinyin/ipa/phonem/g2p references; _encode_text just calls tokenizer.encode). The text — Han characters, English, Pinyin, IPA — is BPE-tokenized as-is and the model was trained to interpret it. Two documented control syntaxes: (1) PINYIN must be space-separated, tone-NUMBERED syllables in pypinyin Style.TONE3 form, i.e. a digit 1-5 appended to each syllable (e.g. 'nin2 hao3'), NOT tone-mark diacritics like 'nín hǎo'. You can mix raw Han text with inline Pinyin (e.g. '您好，请问您来自哪 zuo4 cheng2 shi4？'). To force a proper-noun reading you write its exact numbered-tone Pinyin. (2) IPA must be wrapped in forward slashes /.../ to mark it distinct from normal text (e.g. '/həloʊ, meɪ aɪ æsk wɪtʃ sɪti juː ɑːr frʌm?/'); the v1.0 README suggests DeepPhonemizer (en_us_cmudict_ipa_forward) to produce the IPA. Any combination of Han / English / Pinyin / IPA in one string is allowed. ZHUYIN/Bopomofo is NOT supported anywhere — only numbered Pinyin and slash-IPA. For Taiwan proper-noun tones, supply the Taiwan reading as numbered Pinyin (e.g. 台積電 = 'tai2 ji1 dian4', 企業 Taiwan reading = 'qi4 ye4').

- _Example:_ `我们今天去 tai2 ji1 dian4 参观，CEO said /həloʊ/ to everyone。`

**zh-TW support.** No dedicated Taiwan-accent (zh-TW) capability is documented. Language tags are 'zh' = generic Mandarin and 'yue' = Cantonese; there is no zh-TW / Taiwan code anywhere in the README, config, or 31-language table. Training data is not described as Taiwan-specific and the model most likely defaults to a mainland-leaning Mandarin accent. HOWEVER, the per-syllable numbered-Pinyin control is the practical lever for Taiwan pronunciation: you can force Taiwan lexical/tone readings token-by-token (e.g. 垃圾 'le4 se4' vs mainland 'la1 ji1'; 企業 'qi4 ye4' vs 'qi3 ye4'; 和 'han4'), which is exactly what is needed for thousands of proper nouns. Segmental accent traits (reduced retroflex, no erhua) and Taiwan prosody are NOT separately controllable, though voice cloning from a Taiwan-accented reference clip can carry some of the speaker's accent/timbre. Net: tone/reading control = strong (high confidence); native Taiwan-accent rendering = unsupported/uncertain (low confidence, no primary evidence either way). Recommend an A/B listening test with and without a Taiwan reference voice.

**Code-switch.** Supported within a single utterance. The v1.0 capability list (canonical for v1.5) explicitly lists 'Code-switching — natural mixed-language generation within a single utterance (e.g., Chinese-English), with smooth transitions, consistent speaker identity, and pronunciation-aware rendering on both sides of the switch.' Demonstrated by README example text_5='您好，请问您来自哪 zuo4 cheng2 shi4？' (Han + Pinyin mixed) and the model accepts free mixing of Han/English/Pinyin/IPA in one `text` string; a single language tag (e.g. language='Chinese') is set for the dominant language. Confidence: high that zh+en in one sentence works; exact English-word pronunciation quality inside Chinese context should be spot-checked, and you can wrap tricky English proper nouns in /IPA/ for control.

**Voice mode.** type=mixed, needs_reference_audio=False, needs_reference_text=False.

Three modes via the HF processor: (1) Reference-less default synthesis — build_user_message(text=..., language=...) with NO reference produces audio in a model-chosen voice, so audio CAN be generated with zero reference inputs. (2) Zero-shot voice cloning — pass reference=[audio_url_or_path] (one clip; URL/path/Tensor accepted); this needs ONLY the reference audio, no transcript, because the clip is encoded to audio tokens placed under '- Reference(s):' in the prompt template. (3) Continuation cloning — build_assistant_message(audio_codes_list=[prefix_audio]) with mode='continuation'; this DOES require the prefix transcript prepended to the text. There is NO preset speaker library and NO voice-design-by-text in this checkpoint (instruction/quality fields exist in the template but voice-design-from-text is the separate MOSS-VoiceGenerator model). The SGLang-Omni server additionally lets you pass references[].text and recommends it 'for better speaker similarity', but it is optional in the transformers path. processor.audio_tokenizer = processor.audio_tokenizer.to(device)  # codec must be on device
model = AutoModel.from_pretrained(
    model_id, trust_remote_code=True,
    attn_implementation="sdpa",  # use "flash_attention_2" only if flash-attn is installed
    torch_dtype=dtype,
).to(device).eval()

# zh sentence with an English word; force a proper-noun reading via numbered Pinyin
text = "我们今天去 tai2 ji1 dian4 参观，然后说 hello。"
conv = [processor.build_user_message(text=text, language="Chinese")]
batch = processor([conv], mode="generation")

with torch.no_grad():
    out = model.generate(
        input_ids=batch["input_ids"].to(device),
        attention_mask=batch["attention_mask"].to(device),
        max_new_tokens=4096, do_sample=True,
        audio_temperature=1.7, audio_top_p=0.8, audio_top_k=25,
        audio_repetition_penalty=1.0,
    )

for msg in processor.decode(out):
    if msg is None:
        continue
    audio = msg.audio_codes_list[0]          # STEREO tensor [channels, samples]
    torchaudio.save("out.wav", audio.cpu(), processor.model_config.sampling_rate)  # 48000 Hz
    break

### Qwen/Qwen3-TTS-12Hz-1.7B-Base

- **Repo:** https://github.com/QwenLM/Qwen3-TTS
- **License:** Apache-2.0 (confirmed in HF model card frontmatter, GitHub LICENSE file, and pyproject.toml license={text='Apache-2.0'}). Model weights and code are Apache-2.0 - commercially usable.
- **Runnable:** moderate (confidence: high)

No separate tokenizer download needed (from_pretrained auto-fetches speech_tokenizer/* from the same repo). / 24000 Hz / trust_remote_code=False

**What it is.** Qwen3-TTS-12Hz-1.7B-Base is Alibaba Qwen's open (Apache-2.0) end-to-end autoregressive TTS LM (~1.93B params, BF16). It is a discrete multi-codebook LM: input text is tokenized by a plain Qwen2 BPE text tokenizer (151k vocab, no phoneme frontend) and the talker generates 16-codebook acoustic codes at a 12.5 Hz frame rate, which the bundled Qwen3-TTS-Tokenizer-12Hz codec decodes to a 24 kHz waveform (lightweight non-DiT decoder). The Base variant's only task is 3-second zero-shot voice cloning (and fine-tuning); it has no preset speakers and no instruct/voice-design control (those live in the CustomVoice/VoiceDesign variants). Pronunciation is learned implicitly from raw text + a single coarse per-utterance language tag; there is no pinyin/phoneme/SSML input.

**Phonetic control** (pinyin=❌ zhuyin=❌ phoneme/ipa=❌, confidence=high):

NO phonetic/pronunciation control exists. Read directly from the code: qwen_tts/core/models/processing_qwen3_tts.py (Qwen3TTSProcessor.__call__) does nothing but forward the raw `text` string to the Qwen2 BPE tokenizer (tokenizer_class=Qwen2Tokenizer); there is no g2p, no text-normalization, no pinyin/zhuyin/IPA/phoneme conversion. The prompt builder in qwen3_tts_model.py (_build_assistant_text) just wraps the literal text in <|im_start|>assistant\n{text}<|im_end|>. In modeling_qwen3_tts.py the only pronunciation-relevant conditioning fed to the talker is (a) the text embeddings of those BPE tokens and (b) ONE coarse codec `language_id` token per utterance (chinese=2055). A full repo-wide grep for pinyin/zhuyin/bopomofo/phoneme/ipa/g2p/SSML/tone returns nothing in code. There is no way to force the exact tone of a proper noun. The only (unreliable, undocumented, out-of-distribution) lever a caller has is to rewrite the orthography itself — e.g. substitute homophone Hanzi or split characters — which is not a supported tone-control API and will not reliably hit a specific tone. For thousands of proper nouns needing exact tones, this model offers no deterministic mechanism; you would have to depend on the reference-audio accent and the model's learned reading, or post-filter by ASR.

**zh-TW support.** No dedicated Taiwan-Mandarin (zh-TW) support. get_supported_languages() for Base = [auto, chinese, english, german, italian, portuguese, spanish, japanese, korean, french, russian]; 'chinese' maps to a single codec_language_id token (2055) with no regional variant. A repo-wide grep for taiwan/zh-tw/traditional/繁體/台灣 returns nothing. The only Chinese sub-dialect tags that exist anywhere are beijing_dialect (2074) and sichuan_dialect (2062), and those are exposed only through CustomVoice preset speakers (Dylan/Eric) - not in the Base model and never Taiwan/Min-nan. Practically, the 'chinese'/'auto' tag yields standard (mainland-leaning) Mandarin; a Taiwan accent can only be approximated by cloning a Taiwan-accented reference clip (accent is inherited from ref_audio), and even then tone/pronunciation of specific proper nouns cannot be forced. Evidence: config.json talker_config.codec_language_id; modeling_qwen3_tts.py lines 1830-1834 & 2110-2122; CustomVoice config.json spk_is_dialect.

**Code-switch.** Supported within a single utterance. The text path is a Qwen2 151k-vocab BPE tokenizer that natively encodes both Hanzi and Latin tokens in one string, and the model is a single end-to-end LM with one per-utterance language tag (use language='Chinese' or 'Auto'). Evidence: README Cross-Lingual benchmark table reports strong zh-to-en (2.77) and en-to-zh (4.77) mixed/cross-lingual error rates; README batch examples mix Chinese and English items; the voice-clone example synthesizes English prose mixed with math symbols. **Voice mode.** type=zero-shot-clone, needs_reference_audio=True, needs_reference_text=True.

Default & best-quality path is ICL mode (x_vector_only_mode=False) which ALSO requires ref_text (the reference transcript) - code raises ValueError if ref_text is missing. No preset speakers and no instruct/style/voice-design control in Base (those are in the CustomVoice and VoiceDesign sibling models). Reusable prompts via create_voice_clone_prompt(). ### k2-fsa/OmniVoice

- **Repo:** https://github.com/k2-fsa/OmniVoice
- **License:** Apache-2.0 (both the GitHub code and the HF model weights; pyproject.toml and HF cardData both state apache-2.0). The Higgs-Audio-v2 tokenizer in audio_tokenizer/ carries its own LICENSE file (also permissive). README includes an ethical-use disclaimer prohibiting unauthorized voice cloning/impersonation, but it is not a license restriction.
- **Runnable:** easy (confidence: high)
- **GPU / disk / sample rate:** ~2-4 GB for the TTS model in fp16 (Qwen3-0.6B-class, 612M params; docs report it running on a 4 GB Intel Arc A310). Budget ~5-6 GB if Whisper-large-v3-turbo is also loaded (i.e. when ref_text is not supplied). Any T4/L4/A10/A10G is comfortable. / ~3.3 GB for the OmniVoice repo (model.safetensors 2.45 GB fp32 + Higgs audio_tokenizer ~0.8 GB + tokenizer.json 11 MB; HF usedStorage 3.27 GB). Add ~1.6 GB if Whisper-large-v3-turbo is downloaded for auto-transcription. / 24000 Hz / trust_remote_code=False

**What it is.** OmniVoice is a massively multilingual (646 languages, 581k hrs) zero-shot TTS built on a Qwen3-0.6B LLM backbone (612M params) used as a masked discrete-diffusion ("diffusion language model-style", non-autoregressive) decoder that predicts 8-codebook acoustic tokens, which a Higgs-Audio-v2 tokenizer/decoder reconstructs to 24 kHz audio. Text is fed essentially raw (no g2p) into the Qwen subword tokenizer inside <|text_start|>...<|text_end|>, with style tokens for language and voice-design instruct. It runs via a dedicated `omnivoice` pip package (OmniVoice.from_pretrained / model.generate), supporting voice cloning, voice design (attribute text), and auto-voice modes, plus inline non-verbal tags and pinyin/CMU-phoneme pronunciation overrides.

**Phonetic control** (pinyin=✅ zhuyin=❌ phoneme/ipa=✅, confidence=high):

YES, explicit per-token pronunciation override exists and is the model's headline 'pronunciation correction' feature. There is NO separate g2p/phoneme frontend and no SSML; control is an INLINE TEXT CONVENTION the model learned during training (paper: 'we stochastically replace characters or words with their corresponding phonetic transcriptions, specifically Pinyin for Chinese and phonemes from the CMU pronunciation dictionary for English'). At inference you embed the override directly in the `text` string. CHINESE: replace the target character(s) with UPPERCASE Hanyu Pinyin carrying a trailing tone DIGIT 1-5 (1=high,2=rising,3=low/dipping,4=falling,5/neutral). The uppercase+digit token sits in place of the Han character; surrounding normal characters are unaffected. ENGLISH: put space-separated UPPERCASE CMU/ARPAbet phonemes (with stress digits) inside square brackets, e.g. [B EY1 S]. Both are passed verbatim through OmniVoice.generate(text=...); _combine_text() only strips whitespace/newlines and the Qwen2 tokenizer tokenizes the raw string. There is NO Zhuyin/Bopomofo support and NO IPA support (English uses ARPAbet, not IPA). For your thousands of zh-TW proper nouns you would pre-convert each to UPPERCASE pinyin+tone-digit. Caveat: only single-syllable examples are officially documented; multi-syllable proper-noun spacing (space-separated uppercase syllables) is inferred, not explicitly documented, so it is worth a quick empirical check.

- _Example:_ `这批货物打ZHE2出售后他严重SHE2本了，再也经不起ZHE1腾了。  (Chinese pinyin+tone override; here 折 is forced to zhé/shé/zhē in three contexts) ; English: He plays the [B EY1 S] guitar while catching a [B AE1 S] fish. ; proper-noun extrapolation for 台積電/TSMC: 我在 TAI2 JI1 DIAN4 工作。`

**zh-TW support.** Weak/indirect. There is NO Taiwan-Mandarin accent control: voice-design Chinese dialects are mainland-only (河南话, 陕西话, 四川话, 贵州话, 云南话, 桂林话, 济南话, 石家庄话, 甘肃话, 宁夏话, 青岛话, 东北话 — see omnivoice/utils/voice_design.py and docs/voice-design.md), and the generic 'zh' language ID maps to ISO 'cmn' (Standard Mandarin, 111k hrs) with the demo using Simplified Chinese, so the default accent leans mainland/Putonghua. Taiwan-related entries that DO exist are separate non-Mandarin languages: Min Nan/Hokkien ('nan', only via Tai-lo romanization, no Han chars) and Formosan aboriginal languages (Atayal 'tay', plus tags szy/pwn/trv) — none of these is Taiwan-accent Mandarin. The only realistic path to a Taiwan accent is zero-shot voice cloning with a Taiwan-accent reference clip (and per docs, cross-lingual/cross-accent cloning carries the reference's accent). Tone/pronunciation of individual zh-TW proper nouns can be fixed via the uppercase-pinyin+tone-digit override. Confidence medium (default-accent claim is inferred from data/config, not from an explicit benchmark).

**Code-switch.** Supported. A maintainer states in HF discussion #7 ('Code-switching??'): "Yes, it supports code-switch between any languages," adding that it works best at phrase/sentence granularity rather than isolated single words. This is consistent with the architecture (Qwen subword tokenizer over raw multilingual text, no per-language g2p) and with the README pronunciation examples that freely mix Latin letters inside Chinese text. So a single utterance like '我在 Apple 工作' is expected to work; isolated one-word switches may be less reliable. Evidence: https://huggingface.co/k2-fsa/OmniVoice/discussions/7 ; omnivoice/models/omnivoice.py _combine_text/_tokenize_with_nonverbal_tags (raw mixed-script passthrough). Confidence high on capability, medium on single-word robustness.

**Voice mode.** type=mixed, needs_reference_audio=False, needs_reference_text=False.

Three modes share one generate() API: (1) VOICE CLONING (zero-shot) - requires ref_audio (3-10s, ideally same language as target); ref_text (the reference transcript) is OPTIONAL - if omitted, the model auto-transcribes ref_audio with openai/whisper-large-v3-turbo. Supplying ref_text avoids the extra Whisper download/VRAM and gives more control. (2) VOICE DESIGN - no audio; pass an `instruct` attribute string (gender/age/pitch/whisper/English-accent/Chinese-dialect), comma-separated and combinable. (3) AUTO VOICE - no reference at all, random voice. So audio can be produced with NO reference (auto/voice-design), hence needs_reference_audio=false overall; but for controllable high-quality cloning you DO want ref_audio. Cloning is the most stable/highest-quality mode per the authors.

Optional Qwen3-ForcedAligner-0.6B (timestamps) adds ~1.2-1.3GB disk + VRAM — NOT needed for accuracy scoring, omit it. vLLM backend reserves memory per gpu_memory_utilization (e.g. 0.7). / False
- **Input audio:** Internally fixed to 16000 Hz, MONO, float32 in [-1,1]. The library auto-downmixes multi-channel and auto-resamples any sample rate (librosa). Accepted inputs to transcribe(audio=...): local file path, http(s) URL, base64/data-URL, or (np.ndarray, sr) tuple. Container formats = anything soundfile/librosa decode (wav, flac, mp3, m4a, ogg). Best practice: feed 16kHz mono PCM WAV to avoid resample artifacts. Long audio is handled automatically: max 1200s (20 min) per ASR call, split at low-energy boundaries by split_audio_into_chunks (lossless concat, no overlap); if forced-aligner timestamps are enabled the per-chunk cap drops to 180s. min input 0.5s.

Qwen/Qwen3-ASR-1.7B is an open-source ASR model (Apache-2.0) from the Qwen team, built on Qwen3-Omni. Architecture model_type=qwen3_asr (Qwen3ASRForConditionalGeneration): a 128-mel Whisper-style audio encoder (24 layers, d_model 1024, 30s/480000-sample windows) feeding a Qwen3 2048-dim text decoder. It does language ID + transcription for 30 languages + 22 Chinese dialects, and explicitly benchmarks Taiwan Mandarin (CV-zh-tw WER 3.77, SOTA among the compared models). The OFFICIAL way to run it is the `qwen-asr` PyPI package (repo QwenLM/Qwen3-ASR), which provides both a transformers backend (Qwen3ASRModel.from_pretrained) and a vLLM backend (Qwen3ASRModel.LLM). vLLM also has native day-0 support (`vllm serve Qwen/Qwen3-ASR-1.7B`). The HF repo ships NO custom modeling .py and NO auto_map, so plain AutoModel+trust_remote_code is NOT the intended path; use the qwen-asr package (or transformers >=4.57.x / vLLM, which include Qwen3ASR natively). Two features matter for fair TTS scoring: (1) a free-form CONTEXT string injected as the system prompt to bias toward proper nouns/hotwords (soft, not a mandate); (2) a language hint passed as the canonical English name "Chinese" (there is NO separate "zh-TW"/"Taiwanese" enum — force language="Chinese" and steer Taiwan style via context). For accuracy scoring of mixed zh-TW + English, use a Mixed Error Rate (per-CJK-char + per-English-word tokens) with NFKC + OpenCC script unification + punctuation stripping + English lowercasing, computed via jiwer. Critical caveat: model output script (Traditional vs Simplified) is not guaranteed, so OpenCC-normalize BOTH hypothesis and reference before scoring or CER will be inflated.

**Minimal inference.**
```python
import torch
from qwen_asr import Qwen3ASRModel

model = Qwen3ASRModel.from_pretrained(
    "Qwen/Qwen3-ASR-1.7B",
    dtype=torch.bfloat16,
    device_map="cuda:0",
    # attn_implementation="flash_attention_2",  # optional, Ampere+
    max_inference_batch_size=8,
    max_new_tokens=1024,   # raise for long audio
)

def recognize(wav_path: str, context: str = "", language: str | None = "Chinese") -> str:
    # context = free-form hotword/proper-noun bias (system prompt); language="Chinese" forces zh
    res = model.transcribe(audio=wav_path, context=context, language=language)
    return res[0].text   # res[0].language also available

print(recognize(
    "sample.wav",
    context="台積電 聯發科 護國神山 半導體 NVIDIA GPU CUDA",
    language="Chinese",
))
# Batch: model.transcribe(audio=[w1,w2], context=[c1,c2], language=["Chinese","Chinese"])
```

**Context biasing.** YES — two distinct, independent controls, both exposed on model.transcribe(...). (1) CONTEXT / hotwords: pass context="..." (str, or list aligned to a batch). In the source (qwen_asr/inference/qwen3_asr.py) it is injected verbatim as the SYSTEM message: _build_messages -> [{"role":"system","content":context},{"role":"user","content":[{"type":"audio",...}]}]. The model treats it as background knowledge and SOFTLY tilts decoding toward those tokens — it is a bias, not a hard constraint, so it will not force words that were not spoken. Content can be a short keyword/proper-noun list OR a full paragraph. Practical tips from community sweeps: a labeled phrasing tends to beat a bare space-joined list, e.g. context="專有名詞：台積電、聯發科、PTT、Dcard；Technical terms: CUDA, PyTorch, GPU". Put your TTS prompt's proper nouns / brand names / English tech terms here so they are scored fairly. (2) LANGUAGE hint: pass language=. It is normalized to canonical Title-case and validated against SUPPORTED_LANGUAGES (Chinese, English, Cantonese, ... 30 names). There is NO "zh-TW", "zh", "Taiwanese", or "Mandarin (Taiwan)" enum — passing those raises ValueError. To FORCE Chinese (incl. Taiwan Mandarin) set language="Chinese"; this appends "language Chinese<asr_text>" to the prompt so the model emits text-only in that language (skips auto-LID). For code-switch zh+English, language="Chinese" still transcribes embedded English words correctly (training covers code-switch); alternatively language=None lets the model auto-detect and it returns a merged label like "Chinese,English". For Taiwan flavour specifically: set language="Chinese" AND use context with Taiwan-specific Traditional-script terms; the model has no separate zh-TW switch but benchmarks well on CV-zh-tw. Note: Traditional-vs-Simplified OUTPUT script is not guaranteed by the language hint — handle in scoring (see scoring_recipe). The open-source package has no enable_itn toggle (that exists only on the DashScope cloud API); number formatting in output is therefore model-decided and must be normalized at scoring time.

**Scoring recipe.** Use a Mixed Error Rate (MER): tokenize each CJK character as one token and each contiguous English/alphanumeric run as one word token, then compute Levenshtein WER over those tokens (= CER on Chinese, WER on English, unified). This is the standard zh+en eval (WenetSpeech/ESPnet style). Library: jiwer (jiwer.wer on space-joined token strings). Normalization pipeline, applied IDENTICALLY to reference and hypothesis BEFORE scoring: 1) unicodedata.normalize('NFKC', t) — folds full-width to half-width (digits, latin, punctuation) and removes most width ambiguity; 2) OpenCC script unification — convert BOTH ref and hyp to one script (pick one direction and stick to it, e.g. OpenCC('t2s') to compare in Simplified, or keep Taiwan style with 's2twp'); this is REQUIRED because the model may emit Simplified for zh-TW audio and would otherwise be penalized; 3) English lowercasing t.lower(); 4) strip punctuation (CJK 。，、！？；：「」『』（）《》… and ASCII .,!?;:'\"-()[] ) and collapse whitespace; remove spaces between CJK chars; 5) Number handling: NFKC already unifies width; ASR output and TTS reference may disagree digit-vs-spelled ("3"/"三"/"three"), so EITHER normalize both to a canonical form with a lib like cn2an / num2words, OR document that numbers are scored as-written — pick one and report it. Sketch:\n\n  import re, unicodedata, jiwer\n  from opencc import OpenCC\n  cc = OpenCC('t2s')                      # unify script for BOTH ref & hyp\n  PUNCT = re.compile(r\"[，。！？、；：「」『』（）()\\[\\]{}《》〈〉…—\\-~·.,!?;:\\\"'`]\")\n  CJK   = re.compile(r\"[\\u3400-\\u4dbf\\u4e00-\\u9fff]\")\n  def norm(t):\n      t = unicodedata.normalize('NFKC', t); t = cc.convert(t); t = t.lower()\n      t = PUNCT.sub('', t); return re.sub(r\"\\s+\", ' ', t).strip()\n  def tok(t):\n      out=[]\n      for w in t.split():\n          i=0\n          while i<len(w):\n              if CJK.match(w[i]): out.append(w[i]); i+=1\n              else:\n                  j=i\n                  while j<len(w) and not CJK.match(w[j]): j+=1\n                  out.append(w[i:j]); i=j\n      return out\n  def mer(ref, hyp):\n      return jiwer.wer(' '.join(tok(norm(ref))), ' '.join(tok(norm(hyp))))\n\nReport MER overall, and optionally pure CER (Chinese chars only) and WER (English words only) as a breakdown. For TTS intelligibility, MER between the TTS input text and the ASR transcript = lower is better.

**Modal notes.** Run as a Modal class so the ~5GB model loads once and stays warm. Suggested setup: image = modal.Image.debian_slim(python_version='3.12').pip_install('qwen-asr','soundfile','librosa','opencc-python-reimplemented','jiwer'); add CUDA-matched torch as pulled by qwen-asr. GPU: @app.cls(gpu='L4', ...) (24GB, ample) — A10G/A100 also fine; avoid T4 unless you switch to dtype=torch.float16 and skip flash-attn (Ampere+ only). Cache weights to avoid cold-start downloads: either bake them into the image (run huggingface-cli download Qwen/Qwen3-ASR-1.7B in an image build step) or mount a modal.Volume at ~/.cache/huggingface and download on first run; set HF_HOME to the volume. Load the model in @modal.enter (Qwen3ASRModel.from_pretrained(... device_map='cuda:0')) and call model.transcribe in the method. flash-attn is optional and can be slow/finicky to build — skip it for a first working deploy. Set max_new_tokens=1024+ for long clips. For high-throughput batch scoring of many TTS clips, the vLLM backend (Qwen3ASRModel.LLM or `vllm serve`, install qwen-asr[vllm]) gives much higher concurrency on one GPU; wrap vLLM init under if __name__=='__main__' / inside the Modal container entry to avoid spawn errors. Do NOT load the ForcedAligner unless you need timestamps — it just wastes VRAM for accuracy scoring. Network egress is needed only to download weights and (if you pass URLs) audio; pass local file paths or np arrays for self-contained runs.
